# Emailing the daily briefing (no Windows scheduler needed)

Your morning briefing is generated by Claude and pushed to this GitHub repo. This setup makes
**GitHub Actions email the briefing automatically** — it runs on GitHub's servers, so it does
**not** rely on Windows Task Scheduler (which isn't working on your PC).

**How it works:** every time a new `index.html` is pushed to the `main` branch, the workflow
`.github/workflows/email-briefing.yml` runs, renders the page as the email body, and sends it to
your address. The email therefore always matches the page that was just published.

---

## One-time setup (~5 minutes)

### 1. Create a Gmail App Password (the sender account)
A normal Gmail password won't work for SMTP. You need an App Password, which requires 2-Step
Verification turned on.

1. Go to https://myaccount.google.com/security and turn on **2-Step Verification** (if not already).
2. Go to https://myaccount.google.com/apppasswords
3. Create a new app password (name it e.g. "Briefing emailer"). Google shows a **16-character code** — copy it.

> You can use any Gmail account as the *sender*. If you want the email to come from
> `alexlinda2015@gmail.com`, create the app password on that account.

### 2. Add three secrets to the GitHub repo
On GitHub, open the repo → **Settings → Secrets and variables → Actions → New repository secret**.
Add these three:

| Secret name     | Value                                                            |
|-----------------|------------------------------------------------------------------|
| `MAIL_USERNAME` | the **sender** Gmail address (e.g. `alexlinda2015@gmail.com`)     |
| `MAIL_PASSWORD` | the **16-character App Password** from step 1 (no spaces)        |
| `MAIL_TO`       | the **recipient** (e.g. `alexlinda2015@gmail.com`)               |

Secrets are encrypted by GitHub and are never shown in logs.

### 3. Make sure Actions are enabled
Repo → **Settings → Actions → General** → allow actions to run (default for public repos).

### 4. Test it now
Repo → **Actions** tab → select **Email daily briefing** → **Run workflow** (the
`workflow_dispatch` button). Within ~30 seconds you should receive the current briefing by email.
If it fails, open the run log — the most common cause is a wrong App Password.

---

## Daily flow after setup

1. Claude's scheduled task writes the new `index.html` into this folder each morning.
2. The file is pushed to GitHub. Since Windows Task Scheduler isn't running `auto-push.bat`,
   push it with **one click in GitHub Desktop** (open it, type a summary, **Commit to main** →
   **Push origin** — about 5 seconds). *(If you later fix Task Scheduler, `auto-push.bat` will do
   this automatically and nothing else changes.)*
3. The push triggers GitHub Actions → the briefing lands in your inbox, and GitHub Pages updates.

That's it — the only recurring action is the one-click push, and email + the live web page both
follow from it automatically.

---

## Notes
- **Why not send email from Claude directly?** Claude's task sandbox has no outbound network
  access, and the available email connectors are either draft-only (the Gmail connector can't
  send) or paid (Superhuman). GitHub Actions avoids both problems.
- **Want it on a fixed clock instead of on-push?** Add a `schedule:` (cron) trigger to the
  workflow — but note it would email whatever is currently in the repo, even if that day's push
  hasn't happened yet, so on-push is the safer default.
- **Outlook instead of Gmail?** Change `server_address` to `smtp.office365.com`, `server_port` to
  `587`, and `secure` to `false` in the workflow, and use Outlook credentials.
